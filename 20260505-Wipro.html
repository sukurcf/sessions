Area Paths & Teams

Organizing work and people in GitHub

What Are Area Paths?

Azure DevOps

  • Hierarchical classification for work items
  • Organize by team, feature, or component
  • Each work item belongs to exactly one area
  • Tree structure: Project\Team\Feature
  • Used for security, filtering, and team assignment

GitHub Equivalents

  • Labels — tag issues with team/area/component
  • Custom Fields — single-select fields in Projects
  • Repositories — one repo per component/team
  • Projects — one project per team/initiative
  • Issues can have multiple labels (more flexible)
💡 Key Insight: GitHub doesn't have a single "Area Path" equivalent. Use a combination of labels + custom fields + repo structure for the same organizational power.

Mapping Area Paths to GitHub

🏷️ Strategy 1: Labels

Create labels matching your area paths

  • team:frontend
  • area:payments
  • component:api

Best for: Simple hierarchies

📋 Strategy 2: Custom Fields

Add a single-select "Area" field in Projects

  • Define values for each team/area
  • Group and filter views by field
  • Up to 50 fields per project

Best for: Project-level views

📦 Strategy 3: Repositories

One repo per component or service

  • Natural code + issue separation
  • Cross-repo issues in Projects
  • Team permissions per repo

Best for: Microservices, large orgs

🏗️ Team Features

  • Nested teams — parent/child hierarchy (mirrors area paths)
  • Visibility: Visible (all org members can see) or Secret
  • Roles: Member (default) and Maintainer (manage team)
  • Mentions: @org/team-name notifies all members

⚙️ Team Capabilities

  • Code review assignment — auto-assign PR reviewers
  • Repository access — granular per-repo permissions
  • CODEOWNERS — auto-request reviews by file path
  • Team sync — integrate with identity providers (IdP)

Iterations

Sprint planning and iteration management in GitHub

Iterations in Azure DevOps vs GitHub

Azure DevOps Iterations

  • Hierarchical sprints under project settings
  • Fixed start and end dates
  • Team-specific iteration paths
  • Capacity planning built-in
  • Sprint backlog, burndown charts
  • Velocity tracking across sprints

GitHub Iterations

  • Iteration field in GitHub Projects v2
  • Flexible duration (days, weeks, custom)
  • Supports breaks between iterations
  • Filter with @current, @next, @previous
  • Roadmap view with timeline
  • Insights charts for tracking

Creating Iterations in GitHub Projects

1 Open Project Settings

Navigate to your GitHub Project → click ⋯ menu → Settings → Custom Fields

2 Add Iteration Field

Click New field → select Iteration type → name it (e.g., "Sprint")

3 Configure Iterations

Set start date and duration for each iteration. 3 iterations are auto-created.

4 Add Breaks (Optional)

Insert breaks between iterations for holidays, planning weeks, or PI boundaries

5 Assign Issues

Drag issues to iterations in Board or Table view, or bulk-assign using the field dropdown

6 Create Views

Filter by @current iteration for sprint board. Use Roadmap view for timeline.

📤 Export from Azure DevOps

  • Use ADO REST API to export iteration paths
  • GET /{org}/{project}/_apis/work/teamsettings/iterations
  • Extract: name, start date, end date, path
  • Export to CSV/JSON for mapping

📥 Import to GitHub

  • Use GitHub GraphQL API to create iterations
  • updateProjectV2 mutation for iteration field
  • Script to match dates and names
  • Reassign work items by iteration after import
⚠️ Note: No automated tool migrates iterations directly. Use the APIs to script the migration, or recreate manually if the number of iterations is small.

Providing Access to Users

1 Add to Organization

  • Org Settings → People → Invite
  • Via SAML SSO (Enterprise)
  • SCIM provisioning from Azure AD
  • Enterprise Managed Users (EMU)

2 Add to Team

  • Org → Teams → Select team
  • Click "Add member"
  • Set role: Member or Maintainer
  • Inherits parent team access

3 Set Repo Permissions

  • Team Settings → Repositories
  • Add repos to team
  • Set role: Read/Write/Admin
  • Or per-repo: Settings → Collaborators
💡 Best Practice: Use teams for access control instead of individual user permissions. This mirrors Azure DevOps area path security — assign teams to repos, and all members get the appropriate access level.

Creating, mapping, and migrating work items to GitHub Issues

GitHub Issues: Modern Features

📝 Issue Types GA

Organization-level classification: Bug, Feature, Task, Epic. Up to 25 types per org. Filter and group by type.

🔗 Sub-Issues GA

Parent-child hierarchy. Up to 100 sub-issues per parent. Cross-repo support. Progress tracking built-in.

📊 Issue Fields Preview

Structured metadata directly on issues. Custom fields beyond project-level. Announced March 2026.

🔀 Dependencies New

Define blocks / blocked-by relationships between issues natively.

🏗️ Hierarchy View GA

Browse full sub-issue hierarchy visually within Projects. GA since March 2026.

🔍 Advanced Search GA

AND/OR operators, nested search queries. GA since April 2026. Much more powerful filtering.

Creating Work Items (Issues) in GitHub

🖥️ Via Web UI

  • Repository → Issues → New Issue
  • Select an issue template (Bug, Feature, etc.)
  • Fill title, description (Markdown supported)
  • Set: assignees, labels, milestone, project, iteration
  • Set issue type (Bug, Feature, Task, Epic)
  • Add to Project → set custom fields

⚡ Via GitHub CLI / API

  • gh issue create --title "..." --body "..."
  • --label bug --assignee @user
  • --project "Sprint Board"
  • Bulk create via GraphQL API
  • Import from CSV with scripting
  • Copilot can create issues from Slack

Migration Best Practices

✅ Do

  • Run a pilot migration — test with one project first
  • Map identities — Azure AD → GitHub usernames upfront
  • Keep ADO read-only during transition period
  • Redesign workflows instead of replicating 1:1
  • Train teams on GitHub-specific features
  • Designate champions per team for adoption

❌ Don't

  • Migrate everything at once — use phased approach
  • Expect 1:1 feature parity — GitHub is different by design
  • Skip test migrations — always do dry runs
  • Cut over abruptly — run dual systems briefly
  • Ignore branch policies — recreate as rulesets
  • Forget attachments — export separately from ADO

Key Takeaways

🗂️ Area Paths → Teams + Labels

Use GitHub Teams for people organization and labels/custom fields for work categorization. Teams support nested hierarchy mirroring ADO area paths.

🔄 Iterations → Project Fields

GitHub Projects v2 iteration fields are flexible, support breaks, and integrate with roadmap views. Migrate via API scripting.

🔐 Access → Team-Based Permissions

5-level repo roles (Read through Admin) + team-based access. Use EMU + Azure AD for enterprise-scale identity management.

📋 Work Items → Issues + Projects

GitHub Issues now support types, sub-issues, dependencies, and hierarchy views. Use GEI for repos + ADO Migration Tools for work items.

📌 Remember: GitHub is not a clone of Azure DevOps — it's a different, more integrated platform. Redesign your workflows to leverage GitHub's strengths rather than replicating ADO patterns.
