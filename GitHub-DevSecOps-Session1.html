🏢 Enterprise → Org Flow

  • Go to github.com/enterprises/YOUR_ENTERPRISE
  • Navigate to Settings → Organizations
  • Click "New organization"
  • Enter org name, admin email, and confirm
  • Org inherits enterprise policies (SSO, audit, IP allow list)

⚙️ Post-Creation Setup

  • Configure base member permissions (recommend: Read)
  • Set repo creation policy (restrict to admins)
  • Enable 2FA requirement for all members
  • Create initial teams matching project structure
  • Enterprise SAML SSO applies automatically
💡 Note: Enterprise Owners can create unlimited organizations. Each org inherits the enterprise's authentication, policy, and billing configuration.

👑

Organization-Level Roles (Predefined)

👑 Owner

Complete admin access. Manage members, billing, settings, teams, and all repos. Limit to ≥ 2 people.

👤 Member

Default non-admin role. Can create repos & projects (if allowed). Access via base permissions + teams.

🛡️ Moderator

Block/unblock non-members, set interaction limits, hide comments in public repos.

💰 Billing Manager

Manage billing & payment info. No code or member access.

🔐 Security Manager

View security alerts & manage security settings across all repos. Read access to all repos.

⚙️ GitHub App Manager

Manage GitHub App registrations. Does not grant install/uninstall permissions.

🤝 Outside Collaborator

Repo-specific access only. Not an org member. Called "Repository Collaborator" with EMU.

💡 Custom Roles: Enterprise Cloud orgs can create custom organization roles with granular permissions (e.g., "View audit log" only) without granting full admin access.

📤 Export ADO Users

  • Use ADO REST API: GET _apis/graph/users ?The Azure DevOps Graph API returns all users in the organization. Use continuation tokens to page through large user sets. Endpoint: https://vssps.dev.azure.com/{org}/_apis/graph/users
  • Export: display name, email, UPN, group membership ?UPN = User Principal Name (e.g. user@company.com). This is the primary identifier used to match users between Azure AD and GitHub. Also export group memberships to map to GitHub Teams.
  • Include project-level role assignments ?Capture each user's role per ADO project (Reader, Contributor, Admin). This data drives the GitHub team + repo role mapping.
  • Identify service accounts (exclude from migration) ?Service accounts (build agents, pipeline identities) should NOT be migrated as users. They'll be replaced by GitHub Apps, deploy keys, or machine user accounts in GitHub.

📋 Build Identity Map

  • Create CSV with identity columns based on approach ?EMU approach: Use ado_email, emu_login — the EMU login (shortcode_username) IS the GitHub identity.

    Personal accounts + SSO: Use ado_email, github_handle — map to users' existing GitHub usernames.
  • For EMU: format is shortcode_username ?EMU naming convention: The enterprise shortcode (e.g. "wipro") is prefixed to the username with an underscore: wipro_jsmith. These accounts are fully controlled by the enterprise IdP.
  • Map ADO groups → GitHub teams (1:1 or restructured) ?ADO security groups and project teams should map to GitHub Teams. You can keep a 1:1 mapping or restructure — e.g., flatten nested ADO groups into a simpler GitHub team hierarchy.
  • Flag users needing new GitHub accounts ?Users without existing GitHub accounts need to be provisioned. With EMU, accounts are auto-created by SCIM. With personal accounts, users must create accounts manually and link via SAML SSO.
💡 Output Files: Steps above produce ado-users.json (user export), identity-map.csv (ADO → GitHub mapping), and team-map.csv (group → team mapping). Keep these for the execution phase.

🔵 Azure AD Configuration

  • Add GitHub EMU app from Azure Gallery ?In Azure Portal → Enterprise Applications → New Application → search "GitHub Enterprise Managed User". This pre-built gallery app has SAML + SCIM configuration templates ready to use.
  • Configure SAML SSO: Entity ID, Reply URL, Sign-on URL ?Entity ID: https://github.com/enterprises/SLUG
    Reply URL: https://github.com/enterprises/SLUG/saml/consume
    Sign-on URL: https://github.com/enterprises/SLUG/sso
  • Upload GitHub SAML certificate ?Download the Base64 certificate from Azure AD's SAML config page. Upload it in GitHub Enterprise → Settings → Authentication security → SAML SSO section.
  • Enable SCIM provisioning in the app settings ?SCIM (System for Cross-domain Identity Management) auto-syncs user accounts and group memberships from Azure AD to GitHub. Go to Provisioning tab → Set mode to Automatic.
  • Enter GitHub SCIM endpoint + Bearer token ?Tenant URL: https://api.github.com/scim/v2/enterprises/SLUG
    Secret Token: Generated in GitHub Enterprise → Settings → Authentication → SCIM → Generate token. Store securely.

🟣 GitHub Enterprise Setup

  • Go to Enterprise → Settings → Authentication security ?Navigate to github.com/enterprises/SLUG/settings/auth_security. You need Enterprise Owner role to access this page.
  • Enable SAML single sign-on ?Toggle SAML SSO on. This enforces that all users must authenticate through your IdP (Azure AD) before accessing GitHub resources. Can be set to "required" after testing.
  • Paste Azure AD SSO URL, Issuer, and public cert ?SSO URL: Found in Azure AD → SAML config → Login URL
    Issuer: Azure AD Identifier (Entity ID)
    Public cert: The Base64 X.509 certificate from Azure AD
  • Test SAML configuration with admin account ?Click "Test SAML configuration" in GitHub. This opens a new window that attempts SSO login. Verify it completes successfully before enforcing for all users.
  • Enable SCIM → generate provisioning token ?After enabling SCIM, GitHub generates a one-time Bearer token. Copy it immediately — it won't be shown again. Paste this token into Azure AD's SCIM provisioning configuration.

Phase 4 — Execution & Validation

🚀 Execute Provisioning

  • Assign users/groups to the GitHub app in Azure AD ?In Azure AD → Enterprise Applications → GitHub EMU → Users and groups → Add assignment. Assign individual users or entire AD security groups. SCIM provisioning triggers within ~40 minutes or manually via "Provision on demand".
  • SCIM auto-creates accounts: shortcode_username ?SCIM reads user attributes (displayName, email, userName) from Azure AD and creates matching GitHub accounts with the enterprise shortcode prefix. Example: AD user jsmith → GitHub wipro_jsmith.
  • AD groups auto-map to GitHub Teams ?Azure AD groups assigned to the GitHub app are automatically synced as GitHub Teams. Group membership changes in AD propagate to GitHub via SCIM within the provisioning cycle (~40 min).
  • Users receive email invitation to authenticate ?Once provisioned, users get an email from GitHub to complete their first SSO authentication. They click the link, authenticate via Azure AD, and their GitHub EMU account becomes active.
  • First login via SSO activates the account ?The account exists in GitHub after SCIM provisioning, but remains inactive until the user completes their first SAML SSO login. After activation, they can access repos based on their team assignments.

✅ Validate & Verify

  • Check provisioning logs in Azure AD → Audit logs ?Go to Azure AD → Enterprise Applications → GitHub EMU → Provisioning → Provisioning logs. Filter by status (Success/Failure) to identify any users that failed to provision. Common errors: duplicate emails, invalid characters in usernames.
  • Verify users appear in GitHub Org → People ?Navigate to github.com/orgs/YOUR_ORG/people. All SCIM-provisioned users should appear with their shortcode_username. Check that the total count matches your expected migration scope.
  • Confirm team membership matches AD groups ?Go to github.com/orgs/YOUR_ORG/teams. Each team should mirror the corresponding AD group. Click into each team to verify member lists match. Mismatches usually indicate SCIM group mapping issues.
  • Test SSO login with sample users from each role ?Pick 2-3 users from different roles (admin, contributor, reader). Have them log in via SSO and verify they can access only the repos their team grants. This confirms both SSO and RBAC are working correctly.
  • Validate repo access based on team assignments ?For each team, verify repo permissions match the intended RBAC: Read-only teams should not be able to push, Write teams can create branches and PRs, Admin teams can manage settings. Test with actual Git operations.

🔄 Auto-Provision

Users created automatically when assigned the GitHub app in Azure AD.

📛 Namespaced Accounts

EMU accounts use: shortcode_username. Fully managed by enterprise.

🗑️ Auto-Deprovision

Unassign in AD → user suspended in GitHub. Clean lifecycle management.

Phase 5 — Migration Commands (Step by Step)

1️⃣ Install Tools & Set Up Access

# Install GitHub CLI + ADO2GH extension + Azure CLI
brew install gh && gh extension install github/gh-ado2gh
brew install azure-cli && az login
gh auth login --hostname github.com

# Set environment variables
export ADO_ORG="https://dev.azure.com/yourorg"
export ADO_PAT="your-ado-pat-token"
export GH_ORG="your-github-org"
export GH_PAT="ghp_your-github-pat"
echo $GH_PAT | gh auth login --hostname github.com --with-token

2️⃣ Export ADO Users & Groups

# Export all users → ado-users.json
curl -s -u ":$ADO_PAT" "$ADO_ORG/_apis/graph/users?api-version=7.1" \
  | jq '.value[] | {displayName, mailAddress, principalName, origin}' > ado-users.json

# Export all groups → ado-groups.json
curl -s -u ":$ADO_PAT" "$ADO_ORG/_apis/graph/groups?api-version=7.1" \
  | jq '.value[] | {displayName, descriptor}' > ado-groups.json

# Get members of a specific group
curl -s -u ":$ADO_PAT" \
  "$ADO_ORG/_apis/graph/memberships/$GROUP_DESCRIPTOR?direction=down&api-version=7.1"

3️⃣ Generate Identity Mapping CSV

# Convert ado-users.json → identity-map.csv
cat ado-users.json | jq -r -s '
  ["ado_email","display_name","emu_login","ado_role"],
  (.[] | [.mailAddress, .displayName,
    ("SHORTCODE_" + (.mailAddress | split("@")[0])), "pending"
  ]) | @csv' > identity-map.csv

# Review: remove service accounts, verify emu_login format
ado_emailemu_loginado_role
jsmith@wipro.comwipro_jsmithContributor
apriya@wipro.comwipro_apriyaAdmin
svc-build@wipro.com— SKIP —Service

4️⃣ Build Team Mapping & Create Teams

# Create team-map.csv
echo "ado_group,github_team,github_repo_role" > team-map.csv
echo "ProjectAlpha-Contributors,project-alpha-devs,write" >> team-map.csv
echo "ProjectAlpha-Readers,project-alpha-readonly,read" >> team-map.csv
echo "ProjectAlpha-Admins,project-alpha-admins,admin" >> team-map.csv

# Create teams in GitHub from CSV
while IFS=, read -r ado_group gh_team role; do
  gh api orgs/$GH_ORG/teams -f name="$gh_team" -f privacy="closed" \
    -f description="Migrated from ADO: $ado_group"
done < <(tail -n +2 team-map.csv)

# Assign repos to teams  (push=Write, pull=Read, admin=Admin)
gh api orgs/$GH_ORG/teams/project-alpha-devs/repos/$GH_ORG/my-repo \
  -X PUT -f permission="push"

# Set org base permission to none
gh api orgs/$GH_ORG -X PATCH -f default_repository_permission="none"

5️⃣ Reclaim Mannequin Users (Post-Migration)

# Generate mannequin mapping → mannequins.csv
gh ado2gh generate-mannequin-csv --github-org $GH_ORG --output mannequins.csv

# Edit target_login column in mannequins.csv, then reclaim in bulk
gh ado2gh reclaim-mannequin --github-org $GH_ORG --csv mannequins.csv

Key Takeaways

🏛️ Organization

Create via Enterprise Settings → Organizations. Inherits enterprise SSO, policies, and billing automatically.

📊 Projects

Use Board/Table/Roadmap views. Add custom fields. Automate with built-in workflows and Actions.

🔄 Process

Define issue templates, branch protection, CODEOWNERS, and CI/CD pipelines per repository.

🔒 Permissions

Team-based RBAC. Set org base = No access. Grant via teams. 5 built-in + custom roles (Enterprise).

🎭 RBAC

Read → Triage → Write → Maintain → Admin. Map ADO roles carefully. Least privilege principle.

🚀 User Migration

SAML SSO + SCIM via Azure AD. EMU for managed accounts. Auto-provision & deprovision lifecycle.
