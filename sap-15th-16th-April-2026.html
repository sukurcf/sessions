Use Case 1

PR Workflow Automation

The 3–7 day P90 shipping latency problem across SAP LoBs.

The Problem

  • Multi-repo coordination — CAP + UI5 changes require cross-team reviews
  • Sequential approval gates — Security, architecture, compliance sign-offs
  • CI failures from drift — Environment/config issues mistaken for code bugs
  • Days-long review cycles — 3–7 day P90 shipping latency

The Goal

Build an agentic PR workflow embedded inside GitHub that performs four automated functions on every pull request — summarize, route, triage, and pre-validate.

GitHub Actions Copilot API MCP Server REST/GraphQL
Use Case 1 — Solution

Four Automated Functions

Auto-Summarize PRs

Interpret changes in SAP context — CAP, CDS, OData, UI5 — not generic diffs. E.g., "Adds a new OData entity projection in the CAP CDS model."

Smart Reviewer Routing

Analyze files changed, git blame, CODEOWNERS to recommend and auto-assign the right reviewers instantly.

CI Failure Triage

Fetch logs, parse errors via LLM, post diagnosis + suggested fix. "❌ NullPointerException — OrderDate not initialized in test fixture."

Compliance Pre-Validation

Run security, data privacy, and OSS license checks before any human reviewer sees the PR.

Key Principle: The agent lives where SAP developers already work — PRs, checks, reviews. Context comes from real SAP repos, not generic prompts.

Use Case 1 — Implementation

Implementation Guidance

Trigger & Prompts

  • GitHub Actions workflow triggered on pull_request events (opened, synchronize)
  • SAP-aware prompts: "You are an expert SAP developer assistant familiar with CAP, SAPUI5/Fiori, and OData…"
  • MCP grounding — @cap-js/mcp-server for search_model and search_docs

Design Principles

  • Modular adoption — Each function is a separate workflow step; teams adopt incrementally
  • Calibrate noise — Start with high-confidence outputs, expand iteratively
  • Label AI outputs — All comments authored by SAP-Copilot-Agent

Tradeoff: Automated PR comments save time but create noise if not calibrated. The agent should supplement, not replace, human judgment.

Use Case 2 — Implementation

Building the SRE Agent

Hackathon MVP

  • Start narrow: One incident type — app error logs correlated with recent commits
  • Sample input: Log file with a stack trace + list of recent PRs
  • Expected output: Diagnosis + suggested fix as a new PR
  • Validation cycle: Detection → Suggestion → CI test → Human approval

Use Case 3

Security / Privacy / Compliance PreReview

Shift-Left Governance — finding violations early, not at the gate.

Non-Negotiable Constraints

Secure Coding Standards

Injection, crypto, input validation

GDPR & Tenant Isolation

Data privacy, PII handling, consent

OSS License & Export Controls

License hygiene, dependency risks

Architecture Guardrails

Per-LoB constraints & patterns

Shift Left — Before Human Review

Problem timing: Violations are found late — during formal reviews — creating friction and rework.

Solution: AI prepares artifacts and routes to experts only when needed. Compliance becomes part of the inner loop, not a gate.

Policy knowledge codified in agents, not tribal memory.

Use Case 3 — Solution

Two-Pass Scanning Approach

Pass 1 — Deterministic Analysis

High Confidence • Low Noise
  • Static analysis & linting on code diff
  • Banned functions, missing license headers
  • Known vulnerable dependency versions
  • Rule-based — zero ambiguity

Pass 2 — AI Contextual Analysis

Nuanced • Pattern-Based
  • Data flow patterns — PII exposed across functions
  • Logic bypassing authorization checks
  • Guided prompts: "Check against SAP's security & privacy best practices"
  • Catches what rule-based tools miss

Output: Machine-readable compliance summary (JSON / structured Markdown) categorized by severity — Security, Privacy, Licensing, Architecture — with specific guidance for each finding.

Use Case 3 — Rule Categories

Compliance Rule Checks

Security

No hard-coded credentials, proper input validation, no weak crypto, no dangerous function calls on untrusted input.

Privacy

No logging PII without masking. Data flows respect tenant isolation. External data transfers flagged.

OSS Licensing

Maintain allow-list (MIT, Apache-2.0, BSD) and deny-list. Flag any new dependency not on the allow-list.

Architecture

Layered architecture in CAP (no UI→DB calls), required security headers, approved container registries.

MCP Knowledge Base: Queryable rules via MCP server. Agent checks crypto calls against approved algorithm list retrieved dynamically.

False Positive Mgmt: Start with high-value rules (secrets, critical vulns). Developers can suppress findings with justification. Track accept vs. dismiss for calibration.

Use Case 4 — MCP

MCP-Grounded Context

The @cap-js/mcp-server ensures Copilot writes SAP-correct code from day one.

search_model

Queries the compiled CDS/CSN model for entities, services, fields, and HTTP endpoints. Understands the actual data model before generating code.

search_docs

Searches local CAP documentation embeddings for syntax and best-practice guidance. Validates patterns against official docs.

# AGENTS.md rules for MCP integration
- You MUST search for CDS definitions with cds-mcp,
  only if it fails you MAY read *.cds files directly.
- You MUST search for CAP docs with cds-mcp EVERY TIME
  you create or modify CDS models. Do NOT propose changes
  without first checking it.

Methodology: "Discover first, then change" — ensures SAP-correct output rather than generic guesses.

Cross-Cutting Themes

Common Architectural Patterns

GitHub as System of Record

All four agents operate within GitHub — PRs, Issues, Actions, checks. Insights live where developers already work.

On-Prem Copilot

All analysis stays internal. Proprietary code and production logs never transmitted externally. Security by design.

MCP for Domain Grounding

Model Context Protocol provides SAP-specific knowledge. Agents reason about CAP, CDS, OData — not generic patterns.

Modular Adoption

Each capability is a separate module. Teams adopt incrementally — start with one, expand over time.

Human-in-the-Loop

Agents suggest, humans approve. Clear AI labeling on all outputs. Gradual autonomy for low-risk actions.

Calibrated Sensitivity

Start strict for high-severity, lenient for low. Track accept vs. dismiss. Evolve based on feedback.
